## 1. 项目概览

### 1.1 项目目标

构建"企业合同管理系统"MVP，覆盖合同全生命周期管理：拟制→提交审批→多级审批流转→归档/作废，配套全文检索、模板管理、用户管理、审计日志等功能。

### 1.2 技术栈

| 层 | 技术 |
|----|------|
| 后端框架 | FastAPI (Python 3.11) |
| 前端框架 | Vue 3 Composition API + **TypeScript** + Vite |
| UI 组件库 | Element Plus + **TipTap** 富文本编辑器 |
| 数据库 | SQLite + SQLAlchemy ORM + FTS5（**jieba** 中文分词） |

| 测试 | pytest (44 cases) + Playwright E2E (36 cases) + Python E2E (45 checks) |
| 部署 | Docker Compose + BAT/Shell 一键启动 |

## 3. 做得好的 (What Went Well)

### 3.1 按规格驱动开发

严格遵循 docs-example 的 7 阶段流程：先确立需求基线 → 系统设计 → 阶段门禁审查 → 任务拆分 → 按里程碑推进。每一步都有文档锚定，减少了开发中的摇摆和返工。

### 3.2 SQLite FTS5 全文检索

FTS5 外部内容表 + 触发器同步方案在保持简单架构的同时，提供了英文全文搜索和高亮片段功能。针对中文 CJK 字符的限制，通过 LIKE fallback 提供了实用的折中方案。

### 3.3 审批链引擎

优先级匹配 + JSON 条件/步骤 + 多级顺序流转的设计提供了高度灵活性：
- 条件匹配：金额范围 + 合同类型 + 部门
- 条件之间优先级排序
- 每步指定审批角色 + 超时时长
- 步骤顺序执行，支持驳回和撤回

### 3.4 前后端分离

FastAPI + Vue 3 的组合开发效率高。OpenAPI 自动生成 API 文档（Swagger UI），Vue 3 Composition API + Pinia 状态管理代码清晰。

### 3.5 测试驱动质量保障

- 单元测试使用独立 test.db 隔离
- 测试覆盖了审批流转的正向和异常路径
- E2E 验收脚本覆盖 13 个场景 45 个检查点
- 最终 0 缺陷发布

## 4. 可以改进的 (What Could Be Better)

### 4.1 中文搜索体验

~~FTS5 默认 unicode61 tokenizer 对 CJK 支持有限。当前中文搜索降级为 LIKE `%keyword%` 查询，无高亮。可改进方向：~~
- ✅ **已解决 (FEAT-101)**：集成 jieba 分词 + 应用层分词策略，FTS5 中文搜索精度大幅提升

### 4.2 前端测试缺失

~~当前仅有后端 API 测试，前端缺乏 Cypress/Playwright E2E 测试。在 MVP 阶段可以接受，但后续迭代应补充。~~
- ✅ **已解决 (FEAT-202)**：集成 Playwright 1.61，36 个测试用例覆盖全部 15 个页面，5 套件

### 4.3 审批实例与合同数据耦合

审批实例存储 `snapshot_title` 等字段来避免合同修改影响审批视图，但缺少对原始合同字段变更的完整性校验。

### 4.4 Pydantic V2 迁移的隐性成本

项目使用 Pydantic V2 语法，但在 bcrypt 库兼容性和 FastAPI 生命周期方面踩了坑。这些应在 M0 阶段统一检查依赖兼容性。

### 4.5 无容器化

~~项目没有 Dockerfile/docker-compose，依赖本地 Python 环境和 Node.js。对于演示和分享不够方便。~~
- ✅ **已解决 (FEAT-102)**：Docker Compose 一键部署，前端多阶段构建 + Nginx，数据持久化

### 4.6 合同内容纯文本

~~合同内容使用 `<textarea>`，无法格式化（加粗、列表、标题等），影响合同正文排版。~~
- ✅ **已解决 (FEAT-203)**：集成 TipTap 富文本编辑器（ProseMirror），支持加粗/斜体/标题/列表/引用

### 4.7 数据无法导出

~~合同列表只能在浏览器查看，无法导出为 Excel 或 PDF 进行归档/打印。~~
- ✅ **已解决 (FEAT-204)**：openpyxl (Excel) + weasyprint (PDF)，支持过滤/搜索后导出

### 4.8 前端 JavaScript

~~MVP 前端使用纯 JavaScript，无类型检查，代码可维护性受限。~~
- ✅ **已解决 (FEAT-201)**：TypeScript 渐进式迁移，24 个文件，vue-tsc 0 错误

### 4.9 UI/UX 一致性

~~前端 UI 缺乏统一的设计规范，各页面视觉风格不一致。~~
- ✅ **已解决 (FEAT-205)**：Anthropic Frontend Design — "Precision Enterprise" 设计系统，62 个 CSS Token

## 5. 技术决策回顾

| 决策 | 结果 | 评价 |
|------|------|------|
| SQLite 而非 PostgreSQL | 简化 MVP 部署，FTS5 提供全文检索 | ✅ 正确 — 零配置，满足 MVP 需求 |
| bcrypt 直接使用（不通过 passlib） | 避免 bcrypt 5.x 兼容性问题 | ✅ 正确 — 简洁且无依赖问题 |
| FastAPI lifespan 替代 on_event | 符合最新 FastAPI 最佳实践 | ✅ 正确 |
| Element Plus 而非 Ant Design Vue | 中文文档完善，社区活跃 | ✅ 正确 |
| 不使用 alembic 迁移 | MVP 阶段足够，手动建表 | ✅ 可接受 — 后续应引入 |
| FTS5 外部内容表 | 数据源在 contracts 表，FTS5 索引独立 | ✅ 正确 — 触发器自动同步 |
| 前端无 TypeScript | 2 天 MVP，加速开发 | ✅ 已迁移 (FEAT-201) — vue-tsc 0 错误 |

## 6. 经验教训

### 6.1 需求文档先行

在 M0 之前花时间完善需求文档（MVP 产品基线、SRS、页面与交互说明）和设计文档（系统设计说明书），显著减少了开发过程中的"猜测"和返工。每个页面的交互说明直接指导前端开发，不需要反复确认。

### 6.2 阶段门禁不要跳过

03-阶段门禁的需求与设计一致性审查在 M0 开始前发现了多处不一致（如状态机定义、权限模型），提前修复避免了开发中的偏差。

### 6.3 测试从 M0 开始写

单元测试在 M0 阶段就与代码同步编写（21 个测试覆盖认证 + 合同 + 审批），在 M2 修复阶段发挥了关键作用——每次改动后运行全量测试确保无回归。v1.1/v1.2 迭代中测试从 21 增长到 44（pytest）+ 36（Playwright E2E），形成了 L1 单元测试 + L2 浏览器自动化测试的双层质量防线。

### 6.4 种子数据的重要性

`seed.py` 中的 6 个种子用户 + 4 个示例合同（含已完成审批、进行中审批、已过期等不同状态）让前端开发和演示验证变得高效。v1.1 中新增了 E2E 测试种子用户（e2e_admin/e2e_handler/e2e_approver），数据应覆盖各种状态和边界情况。

### 6.5 API 响应格式应统一

当前 API 响应格式是 `{"data": ..., "message": "..."}`（列表和分页端点），但审批 pending 的 `data` 直接是列表而非对象。E2E 测试脚本编写时需要额外处理这些差异。后续应统一响应格式。

## 7. v1.1 / v1.2 迭代经验（2026-07-09）

MVP 交付后，基于代码债优化成果，一天内完成了 v1.1（6 个功能特性）和 v1.2（5 个质量/UX 特性）的迭代开发。

### 7.1 v1.1：功能增强——补齐 MVP 缺失能力

v1.1 聚焦在 MVP 阶段明确排除但用户感知强烈的特性：jieba 中文分词搜索、Docker 部署、附件上传、站内通知等。在已稳定的 MVP 代码基础上，每个特性平均 30-60 分钟完成。

**关键成功因素：**
- MVP 阶段建立的测试基线（21 tests）让每次改动后都能快速验证无回归
- 代码债优化（15 项任务）先行，为后续特性铺平道路
- 设计文档中明确锁定了每个特性的范围和优先级

### 7.2 v1.2：质量/UX 跃升——从"能用"到"好用"

v1.2 补上了 MVP 阶段遗留的技术债和 UX 短板：TypeScript 迁移 24 个文件、Anthropic Frontend Design 62 个 Token、Playwright 36 个 E2E 测试、TipTap 富文本编辑器、数据导出。

**关键经验：**
- **TypeScript 渐进迁移**：先配置/API → composables/stores → router → 组件，每步验证 build 通过，避免一次性重写风险
- **E2E 测试设计**：API 创建数据 + UI 验证的策略避免了 el-select teleported dropdown 的自动化难题
- **WeasyPrint GTK 依赖**：Windows 环境缺少 GTK3 运行时导致导入失败，改为懒加载 + 优雅降级。跨平台系统依赖是经验盲区
- **TipTap 包体积**：396KB (gzip 126KB)，对 build 影响可控但需持续关注
