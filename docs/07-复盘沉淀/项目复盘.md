## 3. 做得好的 (What Went Well)

### 3.1 按规格驱动开发

严格遵循 docs-example 的 7 阶段流程：先确立需求基线 → 系统设计 → 阶段门禁审查 → 任务拆分 → 按里程碑推进。每一步都有文档锚定，减少了开发中的摇摆和返工。

### 3.2 SQLite FTS5 全文检索

FTS5 外部内容表 + 触发器同步方案在保持简单架构的同时，提供了英文全文搜索和高亮片段功能。针对中文 CJK 字符的限制，通过 LIKE fallback 提供了实用的折中方案。

### 3.3 审批链引擎

优先级匹配 + JSON 条件/步骤 + 多级顺序流转的设计提供了高度灵活性：
- 条件匹配：金额范围 + 合同类型 + 部门
- 条件之间优先级排序
- 每步指定审批角色 + 超时时长
- 步骤顺序执行，支持驳回和撤回

### 3.4 前后端分离

FastAPI + Vue 3 的组合开发效率高。OpenAPI 自动生成 API 文档（Swagger UI），Vue 3 Composition API + Pinia 状态管理代码清晰。

### 3.5 测试驱动质量保障

- 单元测试使用独立 test.db 隔离
- 测试覆盖了审批流转的正向和异常路径
- E2E 验收脚本覆盖 13 个场景 45 个检查点
- 最终 0 缺陷发布

## 4. 可以改进的 (What Could Be Better)

### 4.1 中文搜索体验

FTS5 默认 unicode61 tokenizer 对 CJK 支持有限。当前中文搜索降级为 LIKE `%keyword%` 查询，无高亮。可改进方向：
- 集成 jieba 分词 + 自定义 FTS5 tokenizer
- 或使用 SQLite ICU tokenizer 扩展

### 4.2 前端测试缺失

当前仅有后端 API 测试，前端缺乏 Cypress/Playwright E2E 测试。在 MVP 阶段可以接受，但后续迭代应补充。

### 4.3 审批实例与合同数据耦合

审批实例存储 `snapshot_title` 等字段来避免合同修改影响审批视图，但缺少对原始合同字段变更的完整性校验。可以考虑审批实例创建时全量快照合同内容。

### 4.4 Pydantic V2 迁移的隐性成本

项目使用 Pydantic V2 语法（`ConfigDict`, `model_dump`），但在 bcrypt 库兼容性（bcrypt 5.x 不兼容 passlib）和 FastAPI 生命周期（`@app.on_event` 已废弃）方面踩了坑。这些应在 M0 阶段统一检查依赖兼容性。

### 4.5 无容器化

项目没有 Dockerfile/docker-compose，依赖本地 Python 环境和 Node.js。对于演示和分享不够方便。

## 5. 技术决策回顾

| 决策 | 结果 | 评价 |
|------|------|------|
| SQLite 而非 PostgreSQL | 简化 MVP 部署，FTS5 提供全文检索 | ✅ 正确 — 零配置，满足 MVP 需求 |
| bcrypt 直接使用（不通过 passlib） | 避免 bcrypt 5.x 兼容性问题 | ✅ 正确 — 简洁且无依赖问题 |
| FastAPI lifespan 替代 on_event | 符合最新 FastAPI 最佳实践 | ✅ 正确 |
| Element Plus 而非 Ant Design Vue | 中文文档完善，社区活跃 | ✅ 正确 |
| 不使用 alembic 迁移 | MVP 阶段足够，手动建表 | ✅ 可接受 — 后续应引入 |
| FTS5 外部内容表 | 数据源在 contracts 表，FTS5 索引独立 | ✅ 正确 — 触发器自动同步 |
| 前端无 TypeScript | 2 天 MVP，加速开发 | ⚠️ 可接受 — 后续应迁移 |

## 6. 经验教训

### 6.1 需求文档先行

在 M0 之前花时间完善需求文档（MVP 产品基线、SRS、页面与交互说明）和设计文档（系统设计说明书），显著减少了开发过程中的"猜测"和返工。每个页面的交互说明直接指导前端开发，不需要反复确认。

### 6.2 阶段门禁不要跳过

03-阶段门禁的需求与设计一致性审查在 M0 开始前发现了多处不一致（如状态机定义、权限模型），提前修复避免了开发中的偏差。

### 6.3 测试从 M0 开始写

单元测试在 M0 阶段就与代码同步编写（21 个测试覆盖认证 + 合同 + 审批），在 M2 修复阶段发挥了关键作用——每次改动后运行全量测试确保无回归。

### 6.4 种子数据的重要性

`seed.py` 中的 6 个种子用户 + 4 个示例合同（含已完成审批、进行中审批、已过期等不同状态）让前端开发和演示验证变得高效。数据应覆盖各种状态和边界情况。

### 6.5 API 响应格式应统一

当前 API 响应格式是 `{"data": ..., "message": "..."}`（列表和分页端点），但审批 pending 的 `data` 直接是列表而非对象。E2E 测试脚本编写时需要额外处理这些差异。后续应统一响应格式。
