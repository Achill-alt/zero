### 3.2 FEAT-101 jieba 中文分词搜索 ✅ 已完成

**完成日期：** 2026-07-09

**当前状态：** 已实现 — jieba 分词集成，FTS5 中文搜索精度大幅提升。

**实现方案：**
- **应用层分词策略**：保持 SQLite FTS5 默认 tokenizer（避免自定义 tokenizer 的编译兼容性问题），在应用层用 jieba 对标题/内容进行分词后写入 FTS5 索引
- **写入时**：`create_contract` / `update_contract` 后调用 `reindex_contract()`，将 jieba 分词结果（空格分隔）写入 FTS5，替换原始文本
- **搜索时**：用 jieba 将用户查询分词 → 构建 `"词1" AND "词2" AND "词3"*` 格式的 FTS5 MATCH 查询（AND 语义保证精度，末尾词前缀匹配提升召回）
- **启动时**：`lifespan` 中调用 `reindex_all_contracts()` 回填已有合同的分词索引（idempotent — 每次启动都会重做）
- **回退机制**：保留 LIKE fallback，FTS 无结果时仍用 LIKE 兜底

**涉及文件：**
- `backend/requirements.txt` — 添加 `jieba>=0.42.0`
- `backend/app/services/search_service.py` — 新增 `_segment_text()`, `_build_fts_query()`, `reindex_contract()`, `reindex_all_contracts()`；改造 `search_contracts()`
- `backend/app/services/contract_service.py` — `create_contract` / `update_contract` 调用 `reindex_contract()`
- `backend/app/database.py` — `init_fts5()` 改用 `engine.connect()`（避免 raw sqlite3 连接冲突），支持可选 `engine_` 参数
- `backend/app/main.py` — lifespan 中调用 `reindex_all_contracts()` 回填已有数据
- `backend/tests/conftest.py` — 测试 DB 也初始化 FTS5
- `backend/seed.py` — 创建表后调用 `init_fts5()` 确保触发器就位

**实际实现与计划差异：**

| 计划 | 实际 | 原因 |
|------|------|------|
| 自定义 FTS5 tokenizer | 应用层分词 + FTS5 默认 tokenizer | 自定义 tokenizer 需要编译时启用，标准 Python sqlite3 不支持 |
| 仅 search_service 改造 | 涉及 7 个文件 | FTS5 初始化需重构（raw sqlite3 → engine.connect），seed.py/test 也需适配 |
| `ON CONFLICT DO UPDATE` | `DELETE + INSERT` | FTS5 虚拟表不支持 UPSERT 语法 |

### 3.3 FEAT-102 Docker 容器化部署 ✅ 已完成

**完成日期：** 2026-07-09

**目标：**
- 编写 `Dockerfile`（backend）+ `Dockerfile`（frontend）
- 编写 `docker-compose.yml` 一键编排

**实际交付物：**
```
zero/
├── docker-compose.yml
├── .dockerignore
├── backend/
│   ├── Dockerfile
│   └── docker-entrypoint.sh
└── frontend/
    ├── Dockerfile
    └── nginx.conf
```

**架构：**

```
Browser :5173 → nginx (frontend container)
                    ├── /           → SPA static files (Vue dist)
                    └── /api/*      → proxy → backend:8000 (uvicorn)
                                                     ├── SQLite (./data/app.db)
                                                     └── Uploads (./data/uploads)
```

**涉及文件（新建）：**
- `docker-compose.yml` — 编排 backend:8000 + frontend:80→5173，bridge 网络 `zero-net`
- `.dockerignore` — 排除 `node_modules/`, `__pycache__/`, `.git/`, `*.db`, `logs/` 等
- `backend/Dockerfile` — Python 3.12-slim，pip install → 复制代码 → entrypoint（seed + uvicorn），HEALTHCHECK
- `backend/docker-entrypoint.sh` — seed.py 初始化 DB（idempotent）→ 启动 uvicorn
- `frontend/Dockerfile` — 多阶段构建：node:18-alpine `npm ci` + `vite build` → nginx:alpine 静态服务
- `frontend/nginx.conf` — SPA 路由（`try_files $uri /index.html`）+ `/api/*` 反向代理到 backend:8000 + gzip

**技术决策：**

| 决策 | 选择 | 理由 |
|------|------|------|
| 前端服务 | nginx 反向代理（非 Vite dev server） | 生产级静态文件服务，SPA 路由支持，/api 同源代理无需 CORS |
| 数据持久化 | bind mount `./data:/app/data` | 单目录挂载包含 DB + uploads，简洁清晰 |
| 后端端口 | 8000 同时映射宿主 | 方便直接访问 `/docs` Swagger UI 调试 |
| 前端端口 | 5173（映射容器 80） | 与 BAT 脚本保持一致，用户无需改浏览器书签 |
| Base image | `python:3.12-slim` + `node:18-alpine` + `nginx:alpine` | 最小化镜像体积，alpine 用于前端 |
| Entrypoint | `docker-entrypoint.sh`（seed → uvicorn） | seed.py idempotent，每次启动自动检查/创建 demo 数据 |
| CORS | 无需额外配置 | nginx 反向代理使前后端同源（`localhost:5173`），浏览器不触发 CORS |

**使用方式：**
```bash
# 启动（首次会自动构建镜像）
docker compose up -d

# 查看日志
docker compose logs -f

# 停止
docker compose down

# 停止并清除数据（重新初始化）
docker compose down -v
```

| 适用场景 | Windows 开发机一键启动 | 跨平台部署、CI/CD、生产环境 |
| 日志 | `logs/backend.log` + `logs/frontend.log` | `docker compose logs` |

| 多窗口弹开 | 单窗口，`start /b` 后台运行 | 用户要求单窗口体验 |
| `npm run dev` | `node .../vite/bin/vite.js --host 127.0.0.1` | `.cmd` 包装器在 `start /b` 下不可靠；显式 `--host` 避免 IPv6 问题 |
| `127.0.0.1` | `localhost` + `--host 0.0.0.0`（后端） | 浏览器兼容性 |

**暗坑修复：**
- **CMD 编码**：必须 `chcp 65001` + CRLF 换行符，否则中文乱码 + 命令解析失败
- **嵌套引号**：`start /b "" cmd /c "cd /d \"path\"" ` 引号转义在 CMD 中不可靠 → 先 `cd /d` 再 `start`
- **Vite IPv6**：默认绑定 `localhost` → `::1`，导致 `127.0.0.1:5173` 拒绝连接 → 加 `--host 127.0.0.1`
- **前端空白页**：`el-menu` 的 `router` 属性在 Element Plus 2.14 中导致页面切换后 `<router-view>` 渲染失败 → 改用 `@click="$router.push()"` 显式导航

### 3.6 FEAT-105 站内通知 ✅ 已完成

**完成日期：** 2026-07-09

**决策记录（2026-07-09）：** 原计划为 SMTP 真实邮件通知，经评估后改为**站内通知**。理由：
1. 系统为内部管理工具，用户在工作时间登录使用，站内通知即可满足时效需求
2. 当前 User 模型无邮箱字段，真实邮件需先扩展用户模型
3. MVP 基线将邮件通知明确列为非目标范围，站内通知更符合 MVP 迭代策略
4. 站内通知表结构可平滑升级——后续只需加 SMTP 配置即可扩展为邮件+站内双通道

**实现方案：**
- **数据模型**：`notifications` 表，含 user_id / type / title / content / is_read / related_id / created_at
- **通知类型**：`approval_new`（新审批任务）、`approval_result`（审批结果）
- **创建策略**：按 approver_role 批量创建通知，保证同角色所有审批人可见
- **触发时机**：
  - 提交审批 → 通知第一步骤的审批角色（`contract_service.submit_contract()`）
  - 审批通过（中间步骤）→ 通知下一审批角色（`approval_service.approve_step()`）
  - 审批通过（最后步骤）→ 通知合同创建人（`approval_service.approve_step()`）
  - 审批驳回 → 通知合同创建人（`approval_service.reject_step()`）
- **前端展示**：
  - 导航栏 Bell 图标 + `el-badge` 未读角标，30 秒轮询刷新
  - `el-popover` 下拉面板显示最近 5 条通知，点击跳转合同详情
  - 独立 `/notifications` 页面，分页列表，无通知时 `el-empty` 占位
  - 支持单条已读 + 全部已读操作
